# C4 Model Reference

## Element Types

### Level 1 — System Context

| Element | Definition | Notation |
|---------|-----------|---------|
| `Person` | A human user of the system — internal or external, named by role | Box with person icon |
| `Software System` | The system being described (yours) | Bold-bordered box |
| `External Software System` | Any system outside your boundary that yours calls or is called by | Grey box |

**Rule:** At L1, never show containers, components, or code. The only relationships visible are between People, your system, and external systems.

---

### Level 2 — Container

| Element | Definition | Notation |
|---------|-----------|---------|
| `Container` | A separately deployable/runnable unit: web app, mobile app, desktop app, database, file system, microservice, serverless function, message bus | Box with tech label |
| `ContainerDb` | Database or data store container | Cylinder |
| `ContainerQueue` | Message queue or event bus container | Pipe |
| `External System` | Shown at this level only if it has a direct relationship with a container |  Grey box |

**Rule:** At L2, show what's inside your system boundary. Each container runs in its own process space. A container is NOT a Docker container per se — it is an abstraction for a deployable unit.

---

### Level 3 — Component

| Element | Definition | Notation |
|---------|-----------|---------|
| `Component` | A major structural building block inside a container — a grouping of related functionality with a well-defined interface | Box with component icon |

**Rule:** At L3, zoom into a single container. Show only the components of that container and the external containers/systems they interact with. Do not mix components from different containers in the same L3 diagram.

---

### Level 4 — Code / Class

| Element | Definition | Notation |
|---------|-----------|---------|
| Class / Interface | UML class or interface as generated by IDE or drawn manually | UML class box |

**Rule:** L4 diagrams are optional and should only be produced when the code structure is non-obvious or when onboarding developers. In most cases, the IDE and code-navigation tools replace L4 diagrams.

## Relationship Rules

### Labeling Conventions

Every relationship arrow in a C4 diagram MUST have:
1. **Label** — describes what the caller does or why (e.g., "Places order", not "calls")
2. **Technology** — the protocol or mechanism (e.g., "HTTPS/REST", "gRPC", "JDBC", "Kafka", "WebSocket")

Preferred format: `"[Label]", "[Technology]"`

Example:
```
Rel(spa, apiGateway, "Submits order", "HTTPS/REST")
Rel(orderService, ordersDb, "Reads and writes order data", "JDBC / TCP 5432")
Rel(orderService, eventBus, "Publishes OrderPlaced event", "Apache Kafka")
```

### Synchronous vs Asynchronous

| Relationship Type | Mermaid Style | PlantUML Style | Structurizr |
|------------------|--------------|----------------|-------------|
| Synchronous (request/response) | `-->` solid arrow | `->` | Default `->` |
| Asynchronous (fire and forget) | `-. text .->` dashed | `-->` dashed | Add tag "async" in styles |

In Mermaid C4, use the note in the technology label: "Kafka (async)" to make it explicit.

## Labeling Conventions — Summary

| Element | Name | Description | Technology |
|---------|------|-------------|-----------|
| Person | Role name (e.g., "Customer") | What they do | N/A |
| Software System | System name | One sentence purpose | N/A |
| Container | Functional name | One sentence purpose | Runtime/tech stack |
| Component | Component name | One sentence purpose | Implementation tech |
| Relationship | Verb phrase from caller's perspective | Optional detail | Protocol |

## Common Mistakes

### 1. Mixing Abstraction Levels
**Wrong:** Showing a database container and a class object in the same diagram.
**Fix:** Keep each diagram to one C4 level. If you need to show a class alongside a container, create a separate L4 diagram.

### 2. Missing Technology Stack
**Wrong:** `Container(api, "API Service", "Handles requests")`
**Fix:** `Container(api, "API Service", "Routes and authenticates API requests", "Node.js 20 / Express 4")`

### 3. Unlabeled Arrows
**Wrong:** An arrow between Order Service and Payment Gateway with no label.
**Fix:** `Rel(orderService, paymentGateway, "Processes payment", "HTTPS/REST")`

### 4. Showing Too Much at L1
**Wrong:** Showing 12 microservices at L1 with inter-service arrows.
**Fix:** L1 shows ONE box for your system. Internal decomposition belongs at L2.

### 5. Showing Internal Implementation at L2
**Wrong:** Showing Java classes, database tables, or method calls at L2.
**Fix:** Containers are deployable units. Classes and tables belong at L3/L4.

### 6. Using Person for a Service Account or Batch Process
**Wrong:** Modeling a nightly batch job as a Person.
**Fix:** Use a Container or Component for automated processes. Reserve Person for human actors.

### 7. Not Showing the Direction of Data Flow
**Wrong:** Bidirectional arrows everywhere.
**Fix:** Use directional arrows. If a relationship is truly bidirectional (e.g., WebSocket), label both directions explicitly or add a note.

### 8. Including Every External System
**Wrong:** L1 diagram has 20 external systems, making it unreadable.
**Fix:** At L1, include only external systems that are architecturally significant. Group supporting services (logging, monitoring) into a single "Observability Platform" box if needed.

## Boundary Rules

| Level | System Boundary Shown | What's Inside |
|-------|----------------------|--------------|
| L1 | Your entire software system | Nothing — it's a black box |
| L2 | Your software system | Containers |
| L3 | One container | Components of that container |
| L4 | One component | Classes / interfaces |

At L2 and below, external systems appear outside the boundary and are shown as grey boxes for reference — you do not decompose them.
